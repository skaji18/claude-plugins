# /done: タスク完了スキル

あなたはタスク完了処理を行うスキルです。
ユーザーが依頼元に送信した返信文（または完了報告）を受け取り、該当タスクを特定し、完了処理（done/ への移動 + デイリーログ追記）を実行します。

**処理方針**: スクリプトは使用せず、全てAIとして判断・実行します。

---

## STEP 1: 設定ファイルの読み込み

`~/.task-tracker.json` を Read で読み込み、パスを解決します。

```
CONFIG_PATH = ~/.task-tracker.json
VAULT_PATH  = config.vault_path
SUBFOLDER   = config.subfolder
BASE_DIR    = {VAULT_PATH}/{SUBFOLDER}
INBOX_DIR   = {BASE_DIR}/inbox
DONE_DIR    = {BASE_DIR}/done
DAILY_DIR   = {BASE_DIR}/daily
```

**設定ファイルが存在しない場合**:
以下のメッセージを表示して終了します。他の処理は一切行いません。

```
設定ファイルが見つかりません。
まず /init を実行して初期設定を行ってください。
```

---

## STEP 2: 入力のパース

ユーザーの入力は `$ARGUMENTS` として渡されます。以下の3パターンで解釈します。

### パターン1: 囲い + URL あり

````
/done
https://chat.google.com/room/xxx/thread/yyy

````
LGTM, 2点だけ指摘。
- L42: null check
- L88: typo
````
````

- 囲い外の URL → 返信先の chat URL（参考情報として `## 結果` に記載）
- 囲い内のテキスト → **返信テキスト**（完了処理のメインデータ）

### パターン2: 囲いのみ（URL なし）

````
/done
````
LGTM, 2点だけ指摘。
- L42: null check
- L88: typo
````
````

- 囲い内のテキスト → **返信テキスト**

### パターン3: 囲いなし（自由テキスト）

```
/done
LGTM, 2点だけ軽微な指摘。
- L42: null check抜けてる
- L88: 変数名がtypo
```

- テキスト全体 → **返信テキスト**

### パース手順

1. `$ARGUMENTS` から `/done` プレフィックスを除去（含まれている場合）
2. 4バッククォート（`````）の囲いがあるか判定
   - **あり** → 囲い外と囲い内を分離
     - 囲い外に `https://` で始まる行があれば **chat URL** として記録
     - 囲い内を **返信テキスト** とする
   - **なし** → 全体を **返信テキスト** とする
3. 返信テキストが空の場合 → STEP 3 のタスク特定に進む（返信なしの完了。`## 結果` は「返信なし」と記載）

**ID の事前抽出**:
返信テキストまたは入力全体から `#` + 16進数8桁のパターン（例: `#a3f2b1c8`）を探します。見つかった場合、これを **明示ID** として記録し、STEP 3 の段階1で使用します。

---

## STEP 3: タスク特定ロジック（3段階）

これが /done の核心です。以下の3段階を**上から順に**評価し、最初にマッチした段階で確定します。

### 事前準備: inbox/ のタスク一覧を取得

```
Glob で {INBOX_DIR}/*.md を取得
各ファイルの YAML front matter を Read で読み込む
→ タスクリスト（id, from, type, tags, urls, source, deadline, created, ファイル名, タイトル）
```

**inbox/ が空（0件）の場合**:
以下のメッセージを表示して終了します。

```
完了すべきタスクがありません。
inbox/ にタスクがありません。/add でタスクを登録してください。
```

---

### 段階1: 明示ID による直接特定

**条件**: STEP 2 で明示ID（`#` + 16進数8桁）が抽出されている

**処理**:
1. 明示ID を inbox/ のタスクリストの `id` フィールドと照合
2. **一致あり** → そのタスクに確定。STEP 4 へ進む
3. **一致なし（inbox/ に該当IDがない）** → done/ も検索する
   - done/ に該当IDがある場合:
     ```
     タスク #xxxxxxxx は既に完了済みです。
     done/{ファイル名} に存在します。
     ```
   - どちらにもない場合、inbox/ のタスクリストを候補として提示:
     ```
     ID #xxxxxxxx に一致するタスクが見つかりません。

     現在のタスク:
     1. #a3f2b1c8 PR: auth-refactor レビュー (@田中 / review)
     2. #b7e4d2a1 API設計書レビュー (@鈴木 / doc)

     番号またはIDを指定してください:
     ```

**IDパターンの判定基準**:
- `#` に続く16進数（0-9, a-f, A-F）がちょうど8文字
- 例: `#a3f2b1c8` → 有効、`#abc` → 無効（短すぎ）、`#ghijklmn` → 無効（16進数でない）
- `#` プレフィックスなしの8桁16進数も許容するが、`#` 付きを推奨
  - 例: `a3f2b1c8` も ID として認識するが、通常の英単語と誤判定するリスクがあるため、入力先頭にある場合のみ認識する

---

### 段階2: 1件のみ確認

**条件**: inbox/ のタスクが1件のみ

**処理**:
そのタスクの情報を表示し、確認を求めます。

```
inbox/ にタスクが1件あります:

  #a3f2b1c8 PR: auth-refactor レビュー
  依頼者: 田中さん | 種別: review/pr | 作成: 2026-01-31

このタスクを完了にしますか？

1. はい、このタスクを完了にする
2. いいえ、キャンセルする
```

- **1（はい）** → そのタスクに確定。STEP 4 へ進む
- **2（いいえ）** → 処理を中止。メッセージ:「完了処理をキャンセルしました。」

---

### 段階3: 候補提示（inbox/ に複数タスクがある場合）

**条件**: 段階1・2に該当しない（IDなし かつ 2件以上）

**処理**:

#### 3-A: 関連度によるフィルタリング

返信テキストの内容と inbox/ の各タスクを照合し、関連度を AI として判断します。

**判断の観点**:
- 返信テキスト中の URL が、タスクの `urls` または `source` に含まれるか
- 返信テキスト中のキーワードが、タスクのタイトル・依頼内容・タグに関連するか
- 返信テキストの文脈（レビュー返信ならレビュータスク、設計コメントなら設計タスク等）

**関連度の判定**:
- **明らかに関連** → 候補に含める
- **関連するかもしれない** → 候補に含める（ユーザーが判断する）
- **無関係** → 除外

#### 3-B: 候補の提示

**候補が絞れた場合**（1件以上の関連タスクがある）:

```
返信内容に関連するタスク:

1. #a3f2b1c8 PR: auth-refactor レビュー (@田中 / review)
2. #c5d9e3f7 認証モジュールのテスト (@田中 / implementation)

番号またはIDを指定してください（0: 他のタスクを見る）:
```

**候補が絞れなかった場合**（関連タスクが0件）:

inbox/ の全タスクを `created` が新しい順に最大5件提示します。

```
該当するタスクを特定できませんでした。最近のタスク:

1. #c5d9e3f7 認証モジュールのテスト (@田中 / 2026-01-31)
2. #a3f2b1c8 PR: auth-refactor レビュー (@田中 / 2026-01-30)
3. #b7e4d2a1 API設計書レビュー (@鈴木 / 2026-01-29)
4. #d1f8a6b3 ログ出力改善 (@self / 2026-01-28)
5. #e2c7b4d9 DB移行計画レビュー (@佐藤 / 2026-01-27)

番号またはIDを指定してください（0: 他のタスクを見る）:
```

#### 3-C:「他のタスクを見る」（0 を選択した場合）

5件提示で表示されなかった残りのタスクを全件表示します。

```
全タスク一覧:

1. #c5d9e3f7 認証モジュールのテスト (@田中 / 2026-01-31)
2. #a3f2b1c8 PR: auth-refactor レビュー (@田中 / 2026-01-30)
...（全件表示）
N. #f4a1b2c3 初期設計レビュー (@山田 / 2026-01-10)

番号またはIDを指定してください:
```

---

## STEP 4: 完了処理の実行

タスクが確定したら、以下の4つの処理を順番に実行します。

### 4-1: タイムスタンプの取得

```bash
date "+%Y-%m-%dT%H:%M:%S"
```

この値を `COMPLETED_AT` として記録します。`date` の値を取得した後は以下に進みます。

### 4-2: タスクファイルの更新

対象ファイル: `{INBOX_DIR}/{id}-{slug}.md`

**front matter の更新**:
- `completed: "{COMPLETED_AT}"` を front matter の末尾（`---` の直前）に追記

**本文への追記**:
ファイル末尾に `## 結果` セクションを追加します。

```markdown
## 結果
- 返信: {返信テキストのAI要約（1行、50文字以内）}

### 送信した返信
> {返信テキスト全文（各行を > で引用）}
```

**返信テキストがない場合**（STEP 2 で空だった場合）:
```markdown
## 結果
- 完了（返信なし）
```

**chat URL がある場合**（パターン1）:
```markdown
## 結果
- 返信: {要約}
- 返信先: {chat URL}

### 送信した返信
> {返信テキスト全文}
```

Edit ツールで front matter に `completed` を追記し、Write または Edit で `## 結果` セクションを追加します。

### 4-3: ファイルの移動

```bash
mv "{INBOX_DIR}/{id}-{slug}.md" "{DONE_DIR}/{id}-{slug}.md"
```

**移動失敗時の対処**:
```
エラー: ファイルの移動に失敗しました。
  移動元: {INBOX_DIR}/{id}-{slug}.md
  移動先: {DONE_DIR}/{id}-{slug}.md

手動で以下を実行してください:
  mv "{移動元}" "{移動先}"

タスクファイルの更新（completed の追記、結果セクションの追加）は完了しています。
```

### 4-4: デイリーログの更新

対象ファイル: `{DAILY_DIR}/{YYYY-MM-DD}.md`（`COMPLETED_AT` の日付部分）

**ファイルが存在しない場合** → 新規作成:

```markdown
---
date: "{YYYY-MM-DD}"
---

# {YYYY-MM-DD}

- [x] **{タスクタイトル}** {タグをスペース区切りで列挙}
  - 返信: {返信テキストのAI要約}
  - → [[{id}-{slug}]]
```

**ファイルが既に存在する場合** → 末尾に追記:

```markdown
- [x] **{タスクタイトル}** {タグをスペース区切りで列挙}
  - 返信: {返信テキストのAI要約}
  - → [[{id}-{slug}]]
```

**タスクタイトルの取得**:
タスクファイルの Markdown 本文から `# ` で始まる最初の見出しを取得します。
見出しがない場合は、front matter の `type` + `from` から生成します（例: `review/pr from 田中さん`）。

**タグの取得**:
front matter の `tags` 配列をスペース区切りで列挙します（例: `#proj/backend #type/review`）。

---

## STEP 5: 完了サマリの表示

全ての処理が成功したら、以下のサマリを表示します。

```
✅ タスク完了

  #a3f2b1c8 PR: auth-refactor レビュー
  依頼者: 田中さん
  作成: 2026-01-31T10:30:00 → 完了: 2026-01-31T15:30:00（{経過}）
  返信: LGTM、軽微な指摘2件

  → done/a3f2b1c8-pr-auth-refactor.md
  → daily/2026-01-31.md に記録

  📊 本日の完了: {N}件
```

**経過時間の計算**:
- `created` と `completed` の差分を算出
- 1時間未満 → `{M}分`
- 24時間未満 → `{H}時間{M}分`
- 1日以上 → `{D}日`

**本日の完了件数**:
`{DAILY_DIR}/{YYYY-MM-DD}.md` 内の `- [x]` の行数をカウントします。
ファイルが存在しない場合は `1件`（今回が本日最初の完了）。

## エラーハンドリング一覧

| エラー状況 | メッセージ | 対処 |
|-----------|-----------|------|
| 設定ファイルなし | 「設定ファイルが見つかりません。/init を実行してください」 | 処理中止 |
| inbox/ が空 | 「完了すべきタスクがありません」 | 処理中止 |
| 指定IDが見つからない | 候補を提示して再選択を促す | 段階1の「一致なし」フロー |
| 指定IDが done/ にある | 「タスク #xxx は既に完了済みです」 | 処理中止 |
| ファイル移動失敗 | エラー詳細 + 手動コマンドを提示 | ユーザーに委ねる |
| デイリーログ書き込み失敗 | 「デイリーログの更新に失敗しましたが、タスクの完了処理は完了しています」 | 処理は続行（致命的でない） |
| front matter パースエラー | 「タスクファイルの読み取りに失敗しました: {ファイル名}」 | 該当ファイルをスキップ |
| DONE_DIR が存在しない | `mkdir -p` で自動作成 | 自動復旧 |
| DAILY_DIR が存在しない | `mkdir -p` で自動作成 | 自動復旧 |

---

## 処理フロー全体図

```
STEP 1: Config 読み込み
  ↓ (なければ /init を促して終了)
STEP 2: 入力パース（3パターン判定 + ID事前抽出）
  ↓
STEP 3: タスク特定
  ├─ inbox/ が空 → 終了
  ├─ 段階1: 明示ID → 直接特定
  ├─ 段階2: 1件のみ → 確認して特定
  └─ 段階3: 候補提示 → ユーザー選択で特定
  ↓
STEP 4: 完了処理
  4-1: タイムスタンプ取得 (date コマンド)
  4-2: タスクファイル更新 (completed 追記 + 結果セクション追加)
  4-3: ファイル移動 (inbox/ → done/)
  4-4: デイリーログ更新 (daily/ に追記)
  ↓
STEP 5: 完了サマリ表示
```
